This file is for AI agents and developers working on Kalimba ToneTrace. Keep user-facing explanations in `README.md`; keep implementation and contribution rules here.

## Common Commands

```powershell
npm start
npm run validate:songs
npm run report:rhythm
npm run report:arrangements
npm run build:songs
npm run check
```

Use `npm start` to run the local static server at `http://localhost:8123/index.html`.

## Development Rules

- Prefer small, scoped changes that fit the existing vanilla HTML/CSS/JS app.
- Do not refactor the core audio playback architecture unless the user explicitly asks for that architecture work.
- Do not refactor the microphone pitch-detection architecture unless explicitly asked.
- Do not change the Service Worker cache strategy casually.
- Do not change the UI visual style for documentation or song-data tasks.
- Do not hand-edit `src/songs.js`; it is generated by `scripts/build-song-library.mjs`.
- Add or update built-in songs in `data/songs/*.json`.
- Keep file names and song ids ASCII, lowercase, and hyphenated.
- Keep generated or private research artifacts out of public commits.

## Song Library Workflow

Current built-in melody songs use formal `schemaVersion: 1` for single-note main melodies. The V1 `steps` array is the stable judgeable melody path.

Allowed song contributions:

- PRs may add a new main-melody version.
- PRs may include chord, double-note, or accompaniment information as data-layer extensions when it does not break the single-note melody flow.
- Chord or source details may be recorded in `sourceFeatures`.
- Rich V2 chord arrangements may be discussed, but do not use a song PR as a reason to rewrite playback, microphone, or PWA foundations.

For built-in song changes:

```powershell
npm run validate:songs
npm run build:songs
npm run check
```

If the song includes V2 accompaniment or rhythm windows, also run:

```powershell
npm run report:rhythm
npm run report:arrangements
```

## PWA Version Rules

When runtime resources cached by the Service Worker change, update the app version consistently:

- `package.json` `version`
- `index.html` `appVersionText`
- `songs.html` `appVersionText`
- `src/song-store.js` `APP_VERSION`
- `service-worker.js` `APP_VERSION`
- `CHANGELOG.md`
- `changelog.html`

Runtime resources include files in `APP_SHELL`, generated `src/songs.js`, app JS/CSS, cached HTML pages, icons, manifest, and audio sample manifest. Documentation-only files outside `APP_SHELL` usually do not require a PWA cache version bump.

## Files Not To Hand Edit

- `src/songs.js`: generated from `data/songs/*.json`.
- Browser local storage contents: test by using the app, not by committing local storage dumps.
- Raw generated screenshots or Playwright output directories.

## Directories Not To Commit

Never commit real content from:

- `private/sheets/raw/`
- `private/sheets/ocr/`
- `private/sheets/review/`
- `private/sheets/archive/`
- `private/dev-docs/`

These folders are for source photos, OCR text, review notes, and process records that may contain copyrighted material or local-only research.

## Pre-Release Checks

Before committing:

```powershell
npm run validate:songs
npm run build:songs
npm run check
git status --short
```

Confirm that no files under `private/sheets/raw/`, `private/sheets/ocr/`, or `private/dev-docs/` are staged.

## Practical AI Notes

- Read the current JSON schema and validator before inventing fields.
- Preserve rests as beat gaps; do not create rest notes.
- For accompaniment, avoid filling every beat. Long main-melody gaps of 1 beat or more should usually stay silent unless a reliable source shows held accompaniment.
- Mark uncertain rhythm, chords, or source conflicts in `sourceFeatures` or V2 `rhythm.sourceStatus`.
- Watch for PowerShell console encoding. Use Node or file inspection to verify UTF-8 text if Chinese appears as `????` in terminal output.
